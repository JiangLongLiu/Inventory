## Airtable集成

Airtable集成系统通过AirtableAPI.ts和syncWithAirtable.ts两个核心文件实现。AirtableAPI.ts封装了Airtable REST API的认证和调用，而syncWithAirtable.ts实现了双向数据同步逻辑。

### 认证流程和REST API调用

AirtableAPI类实现了Airtable API的认证和调用机制。该类通过构造函数接收访问令牌(accessToken)和fetch函数，使用Bearer Token认证方式与Airtable API进行通信。

```mermaid
sequenceDiagram
participant Client as "客户端"
participant AirtableAPI as "AirtableAPI"
participant Airtable as "Airtable API"
Client->>AirtableAPI : new AirtableAPI(accessToken)
AirtableAPI->>Airtable : listBases()
Airtable-->>AirtableAPI : 返回bases列表
AirtableAPI-->>Client : 返回bases列表
Client->>AirtableAPI : getBaseSchema(baseId)
AirtableAPI->>Airtable : GET /meta/bases/{baseId}/tables
Airtable-->>AirtableAPI : 返回表结构
AirtableAPI-->>Client : 返回表结构
Client->>AirtableAPI : createRecords(baseId, tableId, records)
AirtableAPI->>Airtable : POST /{baseId}/{tableId}
Airtable-->>AirtableAPI : 返回创建的记录
AirtableAPI-->>Client : 返回创建的记录
```

### 数据同步逻辑

syncWithAirtable.ts实现了与Airtable的数据同步逻辑，采用双向同步策略，确保本地数据和Airtable数据保持一致。同步过程包括推送本地变更到Airtable和从Airtable拉取更新。

```mermaid
flowchart TD
Start([开始同步]) --> PrepareAPI["准备API实例"]
PrepareAPI --> PrepareIntegration["准备集成配置"]
PrepareIntegration --> ProcessSchema["处理Airtable表结构"]
ProcessSchema --> SetSyncVars["设置同步变量"]
SetSyncVars --> SyncCollections["同步Collections"]
SyncCollections --> SyncItems["同步Items"]
SyncItems --> UpdateIntegrationData["更新集成数据"]
UpdateIntegrationData --> End([同步完成])
subgraph "同步Collections"
SyncCollections --> CheckExist["检查记录是否存在"]
CheckExist --> CreateRecords["创建新记录"]
CreateRecords --> UpdateRecords["更新现有记录"]
UpdateRecords --> DeleteRecords["删除标记删除的记录"]
end
subgraph "同步Items"
SyncItems --> CheckExistItems["检查项目记录是否存在"]
CheckExistItems --> CreateItemRecords["创建新项目记录"]
CreateItemRecords --> UpdateItemRecords["更新现有项目记录"]
UpdateItemRecords --> DeleteItemRecords["删除标记删除的项目记录"]
end
```

## Snipe-IT集成

## 扩展指南

为开发者提供扩展新集成服务的指南，包括认证模式、数据映射和错误处理的最佳实践。

### 认证模式

新集成服务应遵循统一的认证模式，使用安全的凭证存储机制，如React Native敏感信息存储。认证信息应在运行时动态获取，避免硬编码。

### 数据映射

数据映射应遵循以下原则：
- 使用类型安全的转换函数
- 处理空值和缺失字段
- 支持双向数据转换
- 验证数据完整性

### 错误处理

错误处理应包括：
- 详细的错误日志记录
- 用户友好的错误消息
- 重试机制
- 断点续传支持
